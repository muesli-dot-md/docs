# S3-compatible storage

Connect an S3-compatible bucket — MinIO, AWS S3, Cloudflare R2, or anything speaking the S3 API — so your documents' canonical `.md` files live in storage your workspace owns.

Storage connections are workspace-scoped and created by a workspace admin; documents are then attached to a connection individually. The live document stays authoritative — the bucket holds its continuously materialized `.md`.

## Credentials

Credentials come from one of two places:

- **Per-workspace keys** — the access key and secret you enter in the connect form are encrypted and stored against the connection. Connections show a `workspace key` badge in Settings.
- **Server environment** — a self-hosted server can instead set credentials once, in its environment; connections created without keys fall back to these and show a `server key` badge:

```sh
MUESLI_S3_ACCESS_KEY=...
MUESLI_S3_SECRET_KEY=...
```

> **Note:** Server-environment secrets never leave the server and are never stored in the database. The connection itself holds only `{endpoint, bucket, region?, prefix?}`.

## Connect a bucket (web app)

1. Open **Settings → Connections → Connected storage**. Connections are per workspace — pick the workspace with the switcher at the top. Only workspace admins can connect or disconnect.
2. Under **Connect storage**, next to **S3 / R2 / MinIO**, click **Attach a bucket**.
3. Fill in **Endpoint URL**, **Bucket**, and optionally **Region** and **Prefix**, plus **Access key ID** and **Secret access key**. After you enter a bucket, the form shows an IAM policy scoped to exactly that bucket and prefix — copy it to create a least-privilege key.
4. Click **Attach**. The server probes the bucket before creating the connection, so a wrong endpoint or bad credentials fail inline instead of breaking sync later.

## Connect and attach over REST

The same flow as two API calls (admin session for the first, editor for the second):

```sh
# 1. Connect a bucket to your workspace
curl -b "$COOKIES" -X POST localhost:8787/api/workspaces/$WS/storage \
  -H 'content-type: application/json' \
  -d '{"kind":"s3","endpoint":"http://localhost:9000","bucket":"muesli-dev"}'
# → {"storage_conn_id":"…"}

# 2. Attach a document; this materializes the current text to the bucket immediately
curl -b "$COOKIES" -X POST localhost:8787/api/documents/my-doc/storage \
  -H 'content-type: application/json' \
  -d '{"storage_conn_id":"…","rel_path":"notes/my-doc.md"}'
```

`rel_path` is optional: it defaults to `<slug>.md`, prefixed by the document's folder path when the document lives in folders.

## How sync behaves

- **Writes:** every edit is written to the object roughly 500ms after the typing burst ends. The server records a SHA-256 of each write in the document's `content_hash`.
- **External changes:** anything that rewrites the object directly is picked up by **polling**, every `MUESLI_STORAGE_POLL_SECS` seconds (default 20; the legacy name `MUESLI_S3_POLL_SECS` still works). The `content_hash` guard keeps muesli from re-ingesting its own writes; genuine external changes are merged into the live document as a text diff.

```sh
MUESLI_STORAGE_POLL_SECS=20
```

> **Note:** Polling latency is expected S3 behavior — external edits appear within one poll interval. If you need instant ingest of local edits, use the [CLI's local-file bridge](local-files.md) alongside or instead.

Sync health for the workspace appears at the bottom of the Connected storage panel — last successful write, or the exact error when a backend is failing.
